  ---
  1. 全量语法扫描 _exhaustive_syntax_scan（第 565 行）

  当 CI 日志中检测到 SyntaxError 时，会对出错文件做迭代式全量扫描：

  # orchestrator.py:584-609
  while True:
      try:
          _ast.parse("\n".join(lines))
          break
      except SyntaxError as e:
          # 记录错误位置
          new_errors.append(ErrorLocation(
              file_path=fp,
              line_number=e.lineno,
              error_type="SyntaxError",
              error_message=str(e),
          ))
          # 注释掉出错行，继续解析下一个错误
          lines[err_lineno - 1] = f"# <<<SYNTAX_ERROR_LINE_{err_lineno}>>>"

  逻辑是：ast.parse() 抛出第一个 SyntaxError → 记录 → 注释掉那行 → 再次 parse →
  发现下一个错误 →
  循环直到全部找到。这样能精准定位每一个语法错误，而不是只报第一个就停。

  ---
  2. 自动填充遗漏文件时的校验（第 796 行）

  在 _validation_gate 中，用 ast.parse() 判断原始文件是否有语法错误：

  # orchestrator.py:796-801
  try:
      ast.parse(orig)
  except SyntaxError:
      logger.warning(f"文件 {fp} 原始内容有语法错误，跳过自动填充（需要修复）")
      continue

  如果原始文件本身就有语法错误，就不会自动填充（保留原内容让 LLM
  修复），避免把错误代码"原样写回"。

  ---
  3. _scan_syntax_files（第 614 行）

  当 CI 日志中 SyntaxError 没有文件路径时，会扫描仓库所有 .py 文件，用
  ast.parse() 逐个检查：

  for file_path in py_files:
      _ast.parse(content)  # 有 SyntaxError 就记录

  ---
  总结

  ast.parse() 解决的核心问题是：CI 
  日志往往只报第一个语法错误就中断了，而一个文件可能有多个语法错误。通过"解析 →
  记录 → 注释出错行 → 再解析"的循环，能一次性找出所有语法错误，避免反复重试。

  每个文件只报第一个语法错误就停了。

  CI 脚本的问题

  你的 CI 语法检查是这样做的：

  python - "$f" << 'PYEOF'
  import ast, sys
  with open(filepath) as fh:
      try:
          ast.parse(fh.read())

  如果一个 .py 文件里有 3 个语法错误，ast.parse() 只会抛出第一个，sys.exit(1)
  直接退出。虽然 continue-on-error: true 让 CI
  继续检查其他文件，但同一个文件里的第 2、第 3 个错误就丢了。

  SpiderClaw 的 ast.parse() 做了什么

  orchestrator.py:584-609 的 _exhaustive_syntax_scan 用了迭代注释的技巧：

  while True:

          ast.parse("\n".join(lines))

          filepath = sys.argv[1]

          while True:
              try:
                  ast.parse("\n".join(lines))
