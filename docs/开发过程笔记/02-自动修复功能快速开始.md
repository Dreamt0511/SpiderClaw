# 自动修复功能快速开始

## 功能概述
SpiderClaw的自动修复功能可以在CI测试失败时，自动分析错误日志、定位问题、生成修复代码、审查代码质量、测试验证，并最终创建Pull Request。

## 系统架构
```
GitHub Webhook → 事件总线 → 主编排器 → 修复Agent → 审查Agent → 测试Agent → 创建PR
```

### 技术栈
| 组件 | 技术选型 | 说明 |
|------|----------|------|
| Agent编排 | LangGraph | 状态机驱动的多智能体流程编排 |
| Agent实现 | LangChain | 标准Agent创建和工具调用框架 |
| LLM | OpenAI / Claude | 支持GPT-4o、Claude 3等大语言模型 |
| Webhook服务 | FastAPI | 高性能异步Web服务 |
| 事件总线 | asyncio.Queue | 本地异步消息队列 |
| Git操作 | GitPython | Git仓库管理和操作 |
| 配置管理 | Pydantic | 类型安全的配置解析 |

### 多Agent架构详解
系统采用三Agent协作架构，每个Agent专注于特定领域：

1. **修复Agent (Fix Agent)**
   - 职责：分析CI错误日志，定位代码问题，生成修复方案
   - 工具：代码读取、搜索、写入等文件操作工具
   - 特点：专注于代码理解和修复生成，确保修复最小化且有效

2. **审查Agent (Review Agent)**
   - 职责：审查修复代码的安全性、质量和合理性
   - 功能：静态安全扫描、变更行数检查、风险评估
   - 特点：严格的安全护栏，防止危险操作和不必要的大变更

3. **测试Agent (Test Agent)**
   - 职责：运行测试用例，验证修复的有效性
   - 功能：自动执行测试、分析测试结果、判断修复是否成功
   - 特点：确保修复不会引入新的错误，保持代码质量

## 前置要求
1. Python 3.11+
2. GitHub个人访问令牌（需要repo权限）
3. OpenAI API密钥（或其他兼容OpenAI API的LLM服务）

## 配置步骤

### 1. 安装依赖
```bash
pip install -r requirements.txt
```

### 2. 配置环境
有两种配置方式：使用配置文件或环境变量。

#### 方式一：使用配置文件
复制示例配置文件：
```bash
cp src/config/agent-config.example.yaml src/config/agent-config.yaml
```

编辑`src/config/agent-config.yaml`，配置以下必填项：
```yaml
# GitHub配置
github:
  token: "ghp_your_github_personal_access_token"

# OpenAI配置
openai:
  api_key: "sk_your_openai_api_key"
  model_name: "gpt-4o"  # LLM模型名称

# Agent配置
agent:
  enabled: true  # 启用自动修复功能
```

#### 方式二：使用环境变量
复制示例环境变量文件：
```bash
cp .env.example .env
```

编辑`.env`文件：
```env
GITHUB__TOKEN=ghp_your_github_personal_access_token
OPENAI__API_KEY=sk_your_openai_api_key
OPENAI__MODEL_NAME=gpt-4o
AGENT__ENABLED=true
```

### 3. 启动服务
```bash
spiderclaw webhook start --secret your_github_webhook_secret

# 使用配置文件启动
python main.py webhook start --config src/config/agent-config.yaml
```

服务启动后会显示：
```
🚀 启动GitHub Webhook服务，访问 http://0.0.0.0:8000/health 检查健康状态
📝 Webhook端点地址: http://0.0.0.0:8000/webhook/github
🤖 自动修复功能已启用
```

### 4. 配置GitHub Webhook
在你的GitHub仓库中设置Webhook：
1. 进入仓库 → Settings → Webhooks → Add webhook
2. Payload URL: `http://your-server-ip:8000/webhook/github`
3. Content type: `application/json`
4. Secret: 与启动服务时使用的secret一致
5. 选择以下事件：
   - Workflow runs
   - Pull requests
   - Check runs
6. 点击"Add webhook"

## 工作流程
当CI测试失败时，系统会自动执行以下步骤：

1. **事件接收**：Webhook收到GitHub的失败事件
2. **上下文收集**：下载CI日志、克隆仓库、解析错误
3. **代码修复**：修复Agent分析错误并生成修复代码
4. **代码审查**：审查Agent检查修复的安全性和质量，确保变更行数≤20行
5. **测试验证**：测试Agent运行测试，验证修复是否有效
6. **创建PR**：如果所有检查通过，自动创建Pull Request

## 功能特性

### 🛡️ 安全护栏
- **危险操作检测**：自动检测并阻止rm -rf、DROP TABLE等危险操作
- **变更行数限制**：默认最多允许修改20行代码，避免过大的变更
- **静态安全扫描**：自动扫描敏感信息泄露、注入漏洞等风险

### 🔄 智能重试
- 最多自动重试3次修复
- 每次失败后根据审查和测试反馈调整修复方案

### 📊 可配置性
- 支持自定义LLM模型（gpt-4o、gpt-3.5-turbo、Claude 3等），通过`openai.model_name`配置
- 支持配置最大重试次数、变更行数限制
- 支持自定义OpenAI API端点（可接入其他兼容OpenAI API的服务）
- 支持配置是否自动创建PR、是否需要人工审批

### 🛠️ 内置工具集
系统内置13个LangChain标准工具，供Agent调用：
- **文件操作**：读取文件、写入文件、搜索代码、获取文件信息
- **Git操作**：克隆仓库、创建分支、提交代码、推送分支、生成diff、创建PR
- **CI工具**：下载CI日志、解析Python错误
- **测试工具**：运行测试用例

## 配置说明

### Agent配置选项
| 配置项 | 说明 | 默认值 |
|--------|------|--------|
| `agent.enabled` | 是否启用自动修复功能 | `false` |
| `agent.max_retries` | 最大修复重试次数 | `3` |
| `agent.max_change_lines` | 最大允许变更行数 | `20` |
| `agent.auto_create_pr` | 是否自动创建PR | `true` |
| `agent.require_human_approval` | 创建PR前是否需要人工审批 | `false` |

### OpenAI配置选项
| 配置项 | 说明 | 默认值 |
|--------|------|--------|
| `openai.api_key` | OpenAI API密钥 | 空 |
| `openai.base_url` | API基础地址 | `https://api.openai.com/v1` |
| `openai.model_name` | LLM模型名称 | `gpt-4o` |
| `openai.timeout` | API请求超时时间（秒） | `60` |

### 支持的CI事件
系统支持三种GitHub事件：
1. **Workflow runs**：CI流水线结束时触发，适合获取整体CI失败信息
2. **Check runs**：CI中单个检查失败时触发，更细粒度的错误信息
3. **Pull requests**：PR状态变化时触发，可针对PR中的代码进行修复

## 最佳实践

### 1. 仓库配置建议
- 在CI配置中启用详细日志输出，有助于Agent更准确地定位错误
- 保持测试用例的完整性，测试覆盖率越高，修复验证越可靠
- 建议在开发环境或测试分支先试用，熟悉系统行为后再用于生产环境

### 2. LLM选型建议
- 对于生产环境，推荐使用GPT-4o或Claude 3 Opus以获得更高的修复准确率
- 测试和开发环境可以使用速度更快、成本更低的模型如gpt-3.5-turbo
- 如果使用自托管LLM，确保模型支持函数调用（Function Calling）功能

### 3. 安全建议
- 定期轮换GitHub访问令牌和OpenAI API密钥
- 限制GitHub令牌的权限范围，只授予必要的repo权限
- 建议开启`require_human_approval`配置，所有PR创建前需要人工审核
- 定期审查自动生成的PR，确保修复符合预期

### 4. 性能优化
- 对于大型仓库，建议配置浅克隆（shallow clone）减少克隆时间
- 配置合适的并发数，避免同时处理过多修复任务导致资源不足
- 定期清理临时目录，避免磁盘空间占用过高

## 系统限制
- 当前版本仅支持Python项目的自动修复
- 单次修复默认最多修改20行代码（可配置）
- 仅支持Git仓库
- 修复效果依赖于CI日志的详细程度和测试用例的覆盖率

## 常见问题

### Q: 系统支持哪些编程语言的修复？
A: 当前版本主要支持Python代码的自动修复，后续会扩展到其他语言。

### Q: 修复准确率如何？
A: 对于简单的语法错误、逻辑错误（如除零、索引越界等），准确率可达80%以上。复杂的业务逻辑错误可能需要人工干预。

### Q: 系统会修改我的生产代码吗？
A: 不会。系统只会创建新的分支和Pull Request，不会直接修改主分支代码。所有变更都需要人工审核后才能合并。

### Q: 如何禁用自动修复功能？
A: 将配置中的`agent.enabled`设为`false`，或在启动时不提供OpenAI API密钥即可。

### Q: 如何获取GitHub个人访问令牌？
浏览器访问：https://github.com/settings/tokens
点击 "Generate new token" → "Generate new token (classic)"
填写 Note（随便写，比如 gh-cli）
勾选这些权限：
✅ repo（全部）
✅ workflow（如果需要操作 Actions）
拉到最下面，点击 "Generate token"
立即复制生成的 Token（ghp_xxxxxxxxxxxx）
🚨 重要提醒：生成之后，页面上的那串以 ghp_ 开头的密钥只会显示这一次。必须立刻复制并保存在安全的地方，关掉页面就再也找不回来了，只能重新生成一个新的。

2. 在终端或代码中使用
拿到令牌后，主要用在两个场景下，用法和密码一样：
命令行操作 Git：当你执行 git clone 或 git push 这类需要权限的操作时，输入用户名后，密码栏直接粘贴这串 Token 就行。为了方便，可以用 git config --global credential.helper store 命令把凭证存起来，省得每次都得输。

在你的代码里调用 GitHub API：千万不要把 Token 硬编码在源代码里，不安全。更规范的做法是把它存到环境变量里（比如 export GITHUB_TOKEN="你的令牌"），然后代码里通过读取环境变量来获取。

把获取到的 Token 写在你本地未提交的那个 src/config/agent-config.yaml 文件里。记得确认文件在 .gitignore 名单里，避免泄露。

## 故障排查

### 1. Webhook接收失败
- 检查服务器端口是否开放，防火墙是否允许8000端口访问
- 验证Webhook Secret是否与配置一致
- 查看GitHub Webhook的最近交付记录，检查错误信息
- 查看服务日志，确认请求是否到达服务器

### 2. 修复流程失败
- 检查GitHub令牌是否有足够的权限（需要repo权限）
- 确认OpenAI API密钥有效且有足够的额度
- 查看服务日志，定位具体失败的阶段（上下文收集/修复/审查/测试/PR创建）
- 验证目标仓库是否可以正常克隆和访问

### 3. 修复效果不佳
- 检查CI日志是否包含足够的错误信息和堆栈跟踪
- 确认测试用例是否能够准确复现问题
- 尝试使用更强大的LLM模型（如GPT-4o）
- 可以在配置中增加`max_retries`次数，允许更多的修复尝试

### 4. 日志查看
```bash
# 查看实时日志
tail -f spiderclaw.log

# 查看错误日志
grep "ERROR" spiderclaw.log

# 查看特定修复流程的日志
grep "event_id: your_event_id" spiderclaw.log
```
