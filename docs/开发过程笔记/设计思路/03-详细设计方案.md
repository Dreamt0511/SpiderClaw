基于 Agent 的服务自动化修复系统——详细设计方案
一、系统概述
1.1 项目定位
构建一个事件驱动的自动诊断与修复系统。当被监控服务发生错误时，系统自动捕获异常事件，通过大模型分析根因，生成修复代码，提交 PR，并通知开发者。

1.2 核心能力
事件驱动监控：不轮询，基于文件监听、Webhook 等机制实时捕获错误

智能根因分析：Agent 从 Traceback 出发，逆向追踪调用链，定位真正根因

跨模块修复：支持同一个服务内跨文件、跨模块的 Bug 追踪与修复

跨服务诊断：结合分布式追踪，定位上游服务传参问题

安全护栏：防止 AI 引入危险代码，确保修复安全可控

多智能体协作：主 Agent 调度，修复 Agent、审查 Agent、测试 Agent 分工协作

二、监控对象与事件源设计
2.1 核心思路
不让 Agent 直接“看服务”，而是让它“看事件”。

系统本质是：事件驱动的自动诊断系统，而非直接监控 Web 服务的运行状态。

2.2 推荐的事件源分类
第一类：日志事件（最核心）
事件类型	说明	触发条件
Python Traceback	未捕获异常	日志中出现 Traceback (most recent call last)
Java Exception	未捕获异常	日志中出现 Exception in thread
Go Panic	运行时崩溃	日志中出现 panic:
Node.js Unhandled Rejection	未处理的 Promise 拒绝	日志中出现 UnhandledPromiseRejection
General Error	通用错误	级别 >= ERROR
日志来源：

日志文件（本地文件系统监听）

stdout/stderr（进程输出捕获）

Docker 容器日志（Docker SDK）

Kubernetes Pod 日志

云平台日志服务（阿里云 SLS 等）

第二类：测试事件
事件类型	说明	获取方式
单元测试失败	pytest/unittest 失败	CI Webhook / 测试输出解析
集成测试失败	跨服务测试失败	CI Webhook
Lint 检查失败	代码风格/静态检查	CI Webhook
类型检查失败	mypy/TypeScript 类型错误	CI Webhook
构建失败	编译/依赖安装失败	CI Webhook
第三类：运行时异常事件
事件类型	说明
进程崩溃	服务进程异常退出
OOM (Out of Memory)	内存耗尽
Restart Loop	服务反复重启
Connection Pool 耗尽	数据库/Redis 连接池满
死锁	数据库死锁
第四类：数据与状态异常
事件类型	说明	Agent 策略
慢查询	SQL 执行时间超阈值	⚠️ 分析 + 建议优化
缓存击穿/雪崩	缓存失效导致流量打到数据库	⚠️ 分析 + 建议
消息堆积	消息队列积压	⚠️ 分析 + 告警
第五类：安全事件
事件类型	说明	Agent 策略
依赖库 CVE	已知安全漏洞	⚠️ 建议升级版本
敏感信息泄露	密钥/密码出现在日志	❌ 告警为主
Agent 能自动修复的优先级：运行时错误 > 测试失败 > 数据异常 > 安全事件

四、错误定位与上下文获取
4.1 核心问题
Traceback 提供调用链，日志提供运行时数据，分布式追踪提供跨服务因果链路。

信息来源	提供内容	获取方式
Traceback	调用链（文件、行号、函数名）	异常捕获 / 日志解析
日志	运行时变量值、请求参数、Trace ID	日志文件 / Webhook
分布式追踪	完整跨服务调用链、上下游请求体	Jaeger/Zipkin API
4.2 Traceback 解析
系统首先从原始日志中提取结构化调用链：

python
# 输入：原始 Traceback 文本
# 输出：结构化调用链
{
    "error_type": "InvalidCardError",
    "error_message": "Card number invalid",
    "call_chain": [
        {"file": "order_controller.py", "line": 120, "function": "create_order"},
        {"file": "order_service.py", "line": 85, "function": "create"},
        {"file": "validator.py", "line": 42, "function": "validate"},
        {"file": "pricing.py", "line": 18, "function": "calculate"}
    ],
    "error_location": {"file": "pricing.py", "line": 18, "function": "calculate"}
}
4.3 跨模块错误追踪（同一服务内）
Traceback 天然包含完整调用链，Agent 接收到后：

读取报错文件代码，理解直接错误

沿调用链向上追溯每一层的逻辑

判断哪一层是根因（而非错误表面位置）

在根因所在位置生成修复

text
示例：
pricing.py:18 报错 ZeroDivisionError（错误表面）
    ↑
validator.py:42 调用 pricing.calculate(items) 时 items 为空（中间层）
    ↑
order_service.py:85 未校验 cart_data 是否为空（根因）

Agent 修复位置：validator.py 加参数校验，而不是在 pricing.py 加 try-except
4.4 跨服务错误追踪
当错误由上游服务引发时，仅凭 Traceback 不够。需要：

方式一：服务注册表（静态配置兜底）
yaml
# service-registry.yaml
services:
  payment-service:
    deploy_path: /app/payment
    repo_url: https://github.com/myteam/payment-service
    production_ref: v2.3.1
    
    # 依赖关系声明
    upstream_services:
      - order-service
      - subscription-service
    
    downstream_services:
      - bank-gateway
    
    # 通知配置
    owners:
      feishu_group: "oc_payment_team"

  order-service:
    deploy_path: /opt/app/order-service
    repo_url: https://github.com/myteam/order-service
    production_ref: v1.1.0
    
    downstream_services:
      - payment-service
      - inventory-service
匹配逻辑：用 Traceback 中的文件路径前缀匹配 deploy_path，找到对应服务及其代码仓库。

方式二：分布式追踪（优先使用）
如果接入了 OpenTelemetry，Agent 用 Trace ID 直接从 Jaeger 获取完整调用链：

python
# Agent 查到上游信息
{
    "upstream_service": "order-service",
    "upstream_operation": "POST /create_order",
    "upstream_request": {"card": "4242", "user_id": 123},
    "current_error": "InvalidCardError"
}
查找上游的优先级策略
策略	准确度	实现成本
分布式追踪 (Jaeger)	⭐⭐⭐⭐⭐	需接入 OpenTelemetry
日志中的 caller_service 字段	⭐⭐⭐⭐	需日志规范
静态配置 (service-registry.yaml)	⭐⭐⭐	需手动维护
推荐：静态配置兜底 + 分布式追踪增强。

4.5 日志等级降级策略
不是所有生产服务都有完整日志，Agent 需根据可用信息逐级降级：

日志质量	Agent 策略	修复质量
完整 Traceback + 变量值	精准定位，能写测试用例复现	⭐⭐⭐⭐⭐
仅有 Traceback	逐层读代码推断，防御性修复	⭐⭐⭐⭐
仅有错误信息	搜索代码库 + Git 历史匹配	⭐⭐⭐
仅有"服务挂了"信号	检查容器状态 + 告警，无法自动修	⭐⭐

六、生产代码与仓库版本处理
6.1 核心原则
生产环境只提供"案发地址"，Agent 在代码仓库中读取和修改代码。

旧代码是线索和起点，不是修复目标。

6.2 版本不一致的处理策略
场景	Agent 行为
Bug 在新代码中已修复	通知"请部署新版本"，不提交代码
Bug 在新代码中仍存在	基于新代码修复
代码重构但 Bug 逻辑还在	追踪文件演变历史，在新代码上修复
紧急热修复（新版本未就绪）	基于生产 Tag 修复，同时 cherry-pick 到主线
6.3 生产与仓库的映射
通过 service-registry.yaml 维护映射关系：

yaml
services:
  order-service:
    deploy_path: /opt/app/order-service    # 生产部署路径
    repo_url: https://github.com/team/order-service
    environments:
      production:
        ref: v1.1.0                      # 生产环境对应的 Git Tag
      staging:
        ref: develop
    test:
      command: "pytest tests/ -v"
