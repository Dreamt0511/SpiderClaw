事件持久化与恢复机制设计                                                                                                                             
                                          
 Context               
                                     
 问题：SpiderClaw 的事件处理依赖内存队列 (asyncio.Queue)，如果服务中断（进程崩溃、重启、部署），队列中未处理的事件会丢失。GitHub 不会重发
 webhook，远程日志端也不会自动重试，导致部分 bug 修复永久丢失。

 目标：在事件进入处理流程前落盘到 SQLite，处理完成后删除；服务重启时根据待处理事件数量动态决定自动恢复或通知开发者确认。

 方案：SQLite 事件队列（方案 A），在端点层落盘。

 ---
 1. 数据模型

 在现有 SQLite (data/repair_records.db) 中新增 pending_events 表：

 CREATE TABLE IF NOT EXISTS pending_events (
     id INTEGER PRIMARY KEY AUTOINCREMENT,
     event_id TEXT UNIQUE NOT NULL,
     event_type TEXT NOT NULL,           -- "github" | "runtime_log"
     payload TEXT NOT NULL,              -- 完整事件 JSON
     status TEXT NOT NULL DEFAULT 'pending',  -- "pending" | "processing"
     source TEXT DEFAULT '',             -- 来源标识（仓库名/服务名）
     created_at REAL NOT NULL,
     updated_at REAL NOT NULL,
     started_at REAL DEFAULT 0,          -- 开始处理时间
     retry_count INTEGER DEFAULT 0
 );

 CREATE INDEX IF NOT EXISTS idx_pending_status ON pending_events(status);
 CREATE INDEX IF NOT EXISTS idx_pending_event_id ON pending_events(event_id);

 - 处理完成后直接 DELETE，repair_records 表已有修复历史
 - started_at 用于检测上次崩溃残留的 'processing' 事件

 ---
 2. 落盘流程

 2.1 事件到达时落盘

 在 webhook_server.py 的两个端点中，publish_event() 之前 INSERT：

 Webhook到达 → [SQLite: INSERT pending] → EventBus入队

 涉及端点：
 - /webhook/github (handle_github_webhook)
 - /webhook/log (handle_log_webhook)

 2.2 事件处理时状态更新

 在 event_consumer() 中：

 consumer取出事件 → [SQLite: UPDATE status='processing', started_at=now]
                          ↓
                    orchestrator.run()
                     ↙          ↘
                  成功            失败
                   ↓               ↓
          [SQLite: DELETE]   [SQLite: UPDATE status='pending']

 2.3 实现位置

 所有落盘逻辑集中在 webhook_server.py：
 - 端点层：INSERT pending
 - consumer 层：UPDATE processing / DELETE / UPDATE pending

 不改动 EventBus、orchestrator.py。

 ---
 3. 恢复流程

 3.1 启动时恢复

 在 run_webhook_server() 中，event_consumer() 启动之前执行：

 服务启动
     ↓
 查询 pending_events WHERE status IN ('pending', 'processing')
     ↓
 重置 status='processing' → 'pending'（上次崩溃残留）
     ↓
 统计待处理数量 N
     ↓
 N ≤ 5？──是──→ 自动重新入队 EventBus，日志记录恢复数量
     │
     否
     ↓
 发送飞书交互式卡片通知
     ↓
 开发者点击"确认恢复" → 重新入队
 开发者点击"放弃" → 删除这些事件

 3.2 阈值配置

 在 agent-config.yaml 新增：

 agent:
   pending_event_auto_threshold: 5  # 自动恢复阈值，超过此数量需人工确认

 3.3 飞书交互式卡片

 标题：⚠️  SpiderClaw 服务恢复通知 — N 个事件待处理

 内容：
 - 事件列表（类型、来源、到达时间）
 - 醒目样式（红色/橙色模板，避免被其他消息淹没）

 按钮：
 - 确认恢复处理 → 回调触发重新入队
 - 放弃这些事件 → 回调触发删除

 回调端点：在 webhook_server.py 新增 /webhook/lark_callback 端点，接收飞书卡片交互回调。

 ---
 4. 去重与防护

 4.1 恢复时去重

 复用现有指纹机制，无需额外逻辑：
 1. replayed 事件进入 orchestrator → collect_context 节点计算 fingerprint
 2. 查 repair_records 表：已修复/已放弃 → 跳过
 3. 失败退避中 → 尊重退避时间

 4.2 防重复落盘

 pending_events 表的 event_id UNIQUE 约束防止同一事件 INSERT 两次。
 INSERT 时使用 INSERT OR IGNORE，静默跳过重复。

 4.3 卡住事件检测

 重启时 status='processing' 的事件视为"上次崩溃残留"，无条件重置为 pending。

 6. 验证方案

 1. 单元测试：PendingEventStore 的 CRUD 操作、UNIQUE 约束、状态重置
 2. 集成测试：
   - 模拟事件到达 → 验证 SQLite 写入
   - 模拟服务重启 → 验证恢复逻辑
   - 模拟处理完成 → 验证 DELETE
   - 模拟处理失败 → 验证回退为 pending
 3. 手动测试：
   - 启动服务 → 发送 webhook → 杀进程 → 重启 → 验证事件被恢复处理
   - 插入 >5 条 pending 事件 → 重启 → 验证飞书通知发送
