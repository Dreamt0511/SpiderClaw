  工作原理：

  业务服务器                          SpiderClaw 服务
  ┌──────────────┐    HTTP POST     ┌──────────────┐
  │ collector.sh │ ──────────────→  │ /webhook/log │
  │ 监控 app.log │   发送错误日志    │  接收并处理   │ 
  └──────────────┘                  └──────────────┘

  部署步骤：

  1. 把 scripts/ 目录复制到业务服务器的 /opt/agent-sidecar/
  2. 编辑 agent-mapping.conf，填写：
    - SERVICE_NAME — 你的服务名称
    - SERVICE_VERSION — 当前版本（Git commit SHA，可选）
    - LOG_PATH — 你要监控的日志文件路径
    - AGENT_URL — SpiderClaw 的 Webhook 地址（如
  http://你的服务器IP:8000/webhook/log）
  1. 启动采集器：
  nohup bash /opt/agent-sidecar/collector.sh > /dev/null 2>&1 &

  脚本会自动做的事：
  - tail -F 实时监控日志文件
  - 检测包含 Error/Traceback/Exception/FAILED 的行
  - 相同错误 300 秒内去重，不重复发送
  - 批量发送（每 10 秒或满 50 行）
  - 429 限流时自动退避重试

所以 SpiderClaw 本身部署在一台服务器上运行 Webhook服务，然后每个需要监控的业务服务器都部署这个采集脚本，把错误日志"推"过来。

## 采集原理

**一句话：`tail -F` 实时追踪日志文件，逐行 grep 错误关键词，攒批后 POST 到 Agent。**

### 采集流程

```
app.log (持续写入)
  │
  ▼
tail -F  ← 实时追踪，文件被 rotate 也能继续读
  │
  ▼
逐行 grep -qiE 'Error|Traceback|Exception|FAILED'  ← 匹配到就存入缓冲区
  │
  ▼
ERROR_CACHE 累积错误行
  │
  ├─ 满 50 行 → 立即发送
  ├─ 距上次发送 ≥ 10 秒 → 发送
  └─ 相同错误 300 秒内 → 去重丢弃
  │
  ▼
curl POST → http://spiderclaw:8000/webhook/log
```

### 发送频率控制

| 参数              | 默认值 | 含义                        |
| ----------------- | ------ | --------------------------- |
| `BATCH_INTERVAL`  | 10 秒  | 攒够 10 秒就发一批          |
| `MAX_BATCH_LINES` | 50 行  | 攒够 50 行立即发            |
| `DEDUP_WINDOW`    | 300 秒 | 同一个错误 5 分钟内不重复发 |

所以**最快 10 秒发一次**，如果错误量大（50 行/批），会更快。

### 去重机制

`compute_hash()` 从错误文本中提取 `File "xxx", line N` + `XxxError`，拼成 key 取 MD5。同一个错误在 300 秒窗口内只发一次。

### 退避重试

收到 429（限流）或非 200 响应时，sleep 翻倍（1→2→4→...→最大 300 秒），收到 200 后重置为 1。

### 发送的数据格式

```json
{
  "log": "完整错误文本（多行）",
  "service": "order-service",
  "version": "abc123",
  "hostname": "biz-server"
}
```

这个 POST 到 `/webhook/log` 后，SpiderClaw 的 Agent 就会拿到错误日志做根因分析和修复。


问题：10秒内就算只有一条错误也会发送吗？

回答：对。看这行判断逻辑：
```bash
if [ $LINE_COUNT -ge "$MAX_BATCH_LINES" ] || \
   ([ $((now - LAST_SEND_TIME)) -ge "$BATCH_INTERVAL" ] && [ $LINE_COUNT -gt 0 ]); then
```

两个条件满足其一就发送：
1. 累积满 50 行 → 立即发
2. 距上次发送 ≥ 10 秒 **且** 缓冲区有内容（`LINE_COUNT > 0`）→ 发

所以**是的，哪怕只有 1 条错误，10 秒到了也会发**。而且第一次错误时 `LAST_SEND_TIME=0`，时间差天然 ≥ 10 秒，所以**第一条错误几乎立即发送**（等 `tail -F` 读完当前行就 flush）。
