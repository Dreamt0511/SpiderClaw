# 二、项目结果展示

## 1）Demo展示

项目实现了开发阶段和线上阶段的Bug监控与自动修复完整闭环。

**开发阶段Demo流程**：开发者向测试仓库提交带有代码错误的PR → GitHub Actions CI自动运行测试并失败 → GitHub Webhook将CI失败事件推送至SpiderClaw → 系统自动拉取CI日志、解析Python异常Traceback → 从Traceback逆向追踪应用代码调用链，定位根因文件和行号 → 多Agent协作生成修复代码 → 通过5重安全验证门禁 → 自动提交修复PR至GitHub → 通过飞书交互式消息卡片通知开发者 → 修复记录自动上报飞书多维表格归档。

**线上阶段Demo流程**：通过docker-compose启动双容器测试环境，biz-server容器模拟真实业务服务运行，内置service_simulator.py模拟8种常见Python错误场景（除零错误、空列表求平均、负数平方根、用户不存在查询、删除不存在用户、连锁异常、折扣率逻辑错误、重复邮箱创建）。collector.sh采集脚本以零侵入方式部署在业务容器中，通过tail -F监控日志文件，匹配到ERROR/Traceback关键词后自动采集上下文，通过HTTP POST上报至SpiderClaw的/webhook/log端点。系统接收错误日志后，经过5秒缓冲窗口合并同服务多个错误，进入事件总线队列，修复编排器依次处理每个错误，自动拉取对应版本的源代码仓库进行针对性修复，生成修复PR并通过飞书通知。

**仪表盘展示**：SpiderClaw提供基于Textual的实时TUI仪表盘，采用暗色海洋夜蓝配色方案。仪表盘分为五个功能模块：左侧事件日志面板按事件类型着色显示（ERROR红色、WARNING橙色、SUCCESS绿色、INFO青色、DEBUG灰色）；右上工具调用面板显示最近12次工具执行状态；右中节点轨迹面板展示LangGraph节点跳转链路，Agent思考节点带有流光动画；右下运行统计面板展示运行时长、Token消耗、LLM调用次数、工具调用次数、修复成功/失败计数；底部系统状态面板展示LLM模型名称、事件队列积压数。仪表盘支持双帧率渲染（Agent思考时30FPS，空闲时20FPS+1s心跳），Windows兼容性良好。

## 2）核心部分代码展示

**Agent核心编排器**（src/agent/orchestrator.py，1900+行）：基于LangGraph的StateGraph构建7节点状态机，包含collect_context、fix_agent、validation_gate、review_changes、run_tests、create_pr、handle_failure节点。通过Command对象实现条件路由，支持最多3次重试和20000 Token预算双控。节点函数返回部分状态更新，不直接修改状态对象。

**错误指纹去重算法**（src/store/repair_store.py）：基于Traceback内容生成稳定的12位hex指纹。策略分四步：排除site-packages、lib/python等框架代码路径；提取应用代码调用链最后3帧（文件名:函数名）；提取异常类型（如ValueError）；提取错误消息前50字符。拼接后MD5取前12位作为唯一标识，配合7种生命周期状态（FIXING、PENDING_DEPLOY、PENDING_PUSH、DEPLOYED、SUPERSEDED、FAILED、ABANDONED）实现精确去重和状态流转。

**5重验证门禁**（src/agent/validation_gate.py）：在修复代码写入文件前执行5重硬校验。导入边界校验剥离所有import行后比较核心代码，语义变更超过阈值则拦截；语法正确性校验通过ast.parse(new_code)确保SyntaxError不引入新错误；文件完整性校验双向检查目标文件是否遗漏及是否修改了非目标文件；错误覆盖校验逐错误位置±3行窗口对比原始与修复代码；变更行数限制仅统计功能性代码变更，排除空行、注释、docstring、导入行。

**强制指令注入系统**（src/agent/instruction_templates.py）：重试时将被拒原因（验证门禁/审查拒绝/测试失败）结构化为强制指令文本，通过system_prompt_override注入System Prompt最高优先级层。不同拒绝来源有不同的指令生成规则：门禁拒绝根据violation_type用模板引擎生成针对性指令；审查拒绝直接传递审查Agent的结构化反馈原文；测试拒绝传递失败用例列表及测试输出。

**事件防丢失持久化机制**（src/store/repair_store.py，770+行）：事件接收即落盘SQLite（INSERT OR IGNORE保证幂等），开始处理标记processing状态，处理成功删除记录，API失败回退为pending状态。服务重启时自动将卡住的processing记录重置为pending。每15分钟后台定时器扫描所有pending事件重新发布到EventBus。推送失败时保存完整上下文（分支名、PR标题、PR正文、diff等），服务启动时自动恢复（recover_pending_pushes）。

**飞书多维表格自动运维**（src/notify/lark_base.py，880+行）：表不存在时自动创建（+table-create），字段缺失时自动补全（+field-create，与预定义字段定义对比差异），日期时间字段格式自动修正（+field-update设置yyyy-MM-dd HH:mm），上报失败自动重试3次（间隔1s）。连续失败超3次触发飞书告警通知，10分钟冷却期防抖。base_token无权限时自动创建新多维表格并持久化到配置文件。

**实时TUI仪表盘**（src/monitor/dashboard/）：手动Alt Screen控制，不依赖rich.live库以解决Windows Git Bash兼容性问题。AuditReader双线程分别tail审计日志文件和应用日志文件，写入线程安全的DashboardState单例，通过signal_refresh触发渲染帧。支持上下箭头滚动日志，按事件类型着色显示。

## 3）项目亮点介绍

**双场景全覆盖**：同时覆盖开发阶段（GitHub Webhook监听CI失败）和生产阶段（Sidecar零侵入采集日志），一套系统解决开发与线上的自动修复需求。

**多Agent协作三级质量管控**：FixAgent负责生成最小化修复代码，ReviewAgent负责两阶段审查（LLM评估+安全自动修复），TestAgent负责代码级验证（ast.parse语法检查+pytest测试执行+import可用性检查）。三级管控确保修复质量，TestAgent完全不消耗LLM Token，降低运营成本。

**飞书生态深度融合**：完整对接飞书消息通知（6种交互式卡片模板含一键跳转按钮）、多维表格（17个字段自动上报修复数据，自动创建和补全表结构）、审批流程（事件积压超阈值自动发起审批，WebSocket长连接实时回调处理结果）、IM命令交互（飞书内发送/status命令实时获取系统运行状态）。

**Sidecar零侵入部署**：生产环境无需修改业务代码，只需要部署一个Shell脚本加一个配置文件。支持日志监控模式（后台tail -F监控日志文件，去重窗口+批量发送+指数退避重试）和命令执行模式（运行命令同时捕获输出，退出码非0或检测到异常时自动上报）。通过spiderclaw init-sidecar命令一键生成采集脚本和配置模板。

**错误分类智能路由**：系统将错误分为三类，不同类别走不同修复路径。A类文件缺失错误（如FileNotFoundError: requirements.txt）直接创建空文件，不调用LLM。B类语法/逻辑错误进行全量AST扫描，迭代注释出错行直到文件完全通过语法解析。C类依赖/导入错误交由FixAgent判断，既可代码修复也可判定为环境问题直接结束流程并通知。

**安全护栏机制完善**：5重后置硬校验防止AI过度修改或引入新错误。仅允许修改src/目录。危险操作自动检测（DROP TABLE、rm -rf、eval、exec、subprocess shell=True等）。单次修复最大修改行数可配置。服务级别修复锁防止同一服务并发修复冲突。

**持久化与重试机制健全**：SQLite存储7种修复生命周期状态，指纹唯一索引保证幂等。失败指数退避重试（60s、120s、240s）。推送失败自动保存完整上下文，后台每10分钟自动重试。事件接收即落盘，服务崩溃不丢失。

## 4）AI 亮点介绍

**高阶AI技巧 - 强制指令系统注入**：重试时将拒绝原因结构化为强制指令文本，通过system_prompt_override注入System Prompt最高优先级层，确保LLM不重复同样的错误。不同拒绝来源有不同的指令生成规则，使修复质量随重试次数递增而非递减。同时使用上下文裁剪策略，当指令累积超过500字符时只保留最新2条，防止Prompt膨胀导致LLM注意力衰减。

**高阶AI技巧 - Token预算双控**：重试决策同时考虑两个维度，重试次数（默认3次）和总Token消耗（20000），任一上限耗尽即停止重试。避免在复杂错误上无限消耗Token，同时提高系统健壮性。

**高阶AI技巧 - 错误指纹与状态机去重**：基于Traceback内容生成稳定指纹，排除框架代码路径干扰，提取应用代码调用链、异常类型、错误消息。配合7种生命周期状态实现精确去重：相同错误正在修复中则跳过，已有修复PR等待部署则跳过，已部署则忽略，修复失败则指数退避重试，失败超3次则放弃。

**高阶AI技巧 - 跨文件根因定位**：从Traceback中的链式错误逆向追踪调用链，支持跨模块关联分析，在根因位置生成修复而非表层打补丁。例如ImportError背后可能隐藏NameError，系统能追踪完整调用链找到真正的根因文件。

**高阶AI技巧 - 错误分类智能路由**：通过规则引擎将错误分为A类（文件缺失）、B类（语法/逻辑错误）、C类（依赖/导入错误），不同类别走不同修复路径。A类不调用LLM直接处理，B类使用AST全量扫描，C类交由FixAgent判断。分类路由显著降低LLM调用次数和Token消耗。

**人与AI的分工**：AI负责异常自动捕获、根因定位、代码修复生成、代码审查（Phase 1评估+Phase 2安全修复）、测试验证（语法检查+测试执行+导入检查）、PR自动创建、飞书通知归档。人负责最终PR Review和Merge决策、服务注册配置、批量恢复审批。核心设计理念是AI做到"提交PR等人审核"，而非直接合入代码，人在最终决策节点保持控制权。

**核心模型选型思路**：通过AgentFactory统一创建FixAgent、ReviewAgent、TestAgent，system_prompt_override机制注入差异化指令。支持OpenAI GPT-4o和Claude通过配置文件切换，LLM参数全部配置驱动。TestAgent完全基于代码执行（ast.parse + subprocess），不消耗LLM Token，降低运营成本。

**AI对工作流的改变**：开发阶段从原来的"提交PR → CI失败 → 人工查日志 → 定位问题 → 修改代码 → 重新提交 → CI再次运行"转变为"提交PR → CI失败 → SpiderClaw自动修复 → 修复PR等待Review → 合并"。线上阶段从"线上报错 → 人工发现 → 拉取日志 → 定位 → 修复 → 发布"转变为"线上报错 → Sidecar自动采集 → SpiderClaw自动修复 → 修复PR等待Review → 合并部署"。研发效率大幅提升，让AI承担重复性排障工作，人专注于需要创造力的决策。

## 5）其他任何信息补充

**技术栈详情**：后端基于Python 3.12+，Agent框架使用LangGraph（LangChain官方状态机编排框架），LLM支持OpenAI GPT-4o和Claude通过配置切换。Webhook服务基于FastAPI + uvicorn，CLI基于Typer + Rich。代码操作基于GitPython避免shell注入风险。测试框架使用pytest + pytest-asyncio。代码质量检查使用pylint + bandit。仪表盘基于Textual自定义实现，手动Alt Screen控制兼容Windows环境。日志系统使用structlog实现结构化输出（JSONL格式+控制台彩色显示）。

**项目规模**：项目共61个Python源文件，核心编排器单文件1900+行，Webhook服务单文件1500+行，飞书多维表格客户端单文件880+行。全量Python类型注解，Pydantic v2运行时校验。

**灵活配置**：所有配置通过agent-config.yaml文件驱动，包含GitHub配置（API Token、允许仓库列表）、LLM配置（API Key、模型名、Base URL）、飞书配置（应用凭证、通知用户、多维表格Token）、Agent配置（修复开关、重试次数、变更行数限制）等11个子模型。

**执行顺序建议**：测试时先推送分支到GitHub，优先使用GitHub CLI创建PR触发CI测试，如果CLI不可用再使用GitHub API作为备用方案。

# 三、其他信息（可选）- 自由发挥区

**设计的初衷与故事**：SpiderClaw的名字来源于蜘蛛的捕食行为——蜘蛛织网后静待猎物触网，然后迅速出击。这正是系统的设计哲学：事件驱动、静默监听、自动响应。与Sentry等传统APM工具不同，SpiderClaw不只是"发现"错误，而是像蜘蛛一样主动"捕获"并"消化"错误——从发现到根因分析再到修复PR提交，全程自动化。

**为什么选择飞书生态**：飞书在国内企业协作市场占有率高，其开放平台提供了完善的消息卡片、多维表格、审批流程等能力。将SpiderClaw深度对接飞书生态，使得修复通知不再是冷冰冰的文本消息，而是带有交互式按钮的富媒体卡片，修复数据自动沉淀为多维表格方便统计分析和趋势追踪，审批流程与飞书审批中心无缝集成。这种深度整合让整个修复流程在企业IM生态内完成闭环，无需额外开发管理后台。

**可复用与可推广性**：SpiderClaw的架构设计具有高度可扩展性。通过扩展代码解析器和Prompt模板，可以支持Java、Go、JavaScript等多语言。通过实现新的事件源适配器，可以接入Sentry、Prometheus、ELK等告警事件源。基于修复历史数据的Fine-tuning可以提升特定场景的准确率。这些扩展点使得SpiderClaw不只是一个工具，更是一个可生长的自动修复平台。

**未来方向**：扩展多语言支持是当前最迫切的路线，Java和Go在企业级应用中占比极高。接入Sentry和Prometheus等告警事件源可以让系统覆盖更多监控场景。基于积累的修复历史数据进行模型微调，可以显著提升特定场景的修复准确率。项目目前仍处于早期阶段，存在一些未完善的功能和性能问题，欢迎社区贡献。
