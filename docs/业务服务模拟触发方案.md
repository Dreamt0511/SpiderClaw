以下是完整的业务服务报错模拟方案：

---

## 方案概览

你已经在 Docker Compose 中跑着 `biz-server` + `collector.sh` + `SpiderClaw Agent`，数据流已经通了：

```
模拟器 → app.log → collector.sh → POST /webhook/log → EventBus → RepairOrchestrator
```

我补充了三个文件，让这个链路完整跑起来：

### 新增文件

| 文件                                             | 作用                       |
| ------------------------------------------------ | -------------------------- |
| `docker/biz-server/app/src/service_simulator.py` | 业务服务模拟器（核心）     |
| `docker/biz-server/app/tests/run_ci_trigger.sh`  | 容器内一键触发所有错误场景 |
| `tests/test_biz_error_trigger.sh`                | 宿主机上控制触发测试的入口 |
| 修改 `docker/biz-server/entrypoint.sh`           | 支持自动启动模拟器         |

---

### 模拟器支持的 8 个错误场景

```
divide_by_zero          — ZeroDivisionError（除零）
empty_average           — ZeroDivisionError（空列表求平均）
negative_sqrt           — ValueError（负数平方根）
user_not_found          — KeyError（查询不存在用户）
delete_nonexistent_user — KeyError（删除不存在用户）
discount_negative_rate  — 逻辑错误（不抛异常，但结果错误）
create_duplicate_email  — 逻辑错误（重复 email 无校验）
chain_reaction          — 连锁异常（除零+空列表先后触发）
```

---

### 三种触发方式

**方式一：一键触发所有场景（适合快速验证）**
```bash
bash tests/test_biz_error_trigger.sh all
```
这会在容器内依次执行 6 个会崩溃的场景，collector 捕获后上报给 Agent。你可以切另一个终端 `docker logs spiderclaw-agent -f` 观察修复流程。

**方式二：持续随机触发（适合长时间测试）**
```bash
bash tests/test_biz_error_trigger.sh continuous 60
```
每 60 秒随机触发一个错误场景，模拟真实业务服务间歇性报错。

**方式三：通过环境变量在容器启动时自动运行**
```bash
# 在 docker-compose.yml 的 biz-server 环境变量中加：
SIMULATOR_MODE=continuous
SIMULATOR_INTERVAL=30
SIMULATOR_ERROR_RATE=0.3
```
重启后 biz-server 会自动循环执行业务操作，按概率触发 bug。

---

### 单独触发某个场景

```bash
# 从宿主机
bash tests/test_biz_error_trigger.sh divide_by_zero
bash tests/test_biz_error_trigger.sh user_not_found

# 或直接进容器
docker exec biz-server bash /opt/biz-app/app/tests/run_ci_trigger.sh empty_average
```

---

### 验证链路是否通畅

```bash
# 1. 检查 Agent 是否正常
curl http://localhost:8000/health

# 2. 触发一个错误
bash tests/test_biz_error_trigger.sh divide_by_zero

# 3. 观察 Agent 是否收到并开始修复
docker logs spiderclaw-agent --tail 50
```

Agent 日志应出现 `接收运行时日志: service=order-service` → `开始修复流程` → `修复完成` 的流程。
